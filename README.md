# System Story Coverage Demo

System Story Coverage is a lightweight technique for connecting schema fields
and enum values to real business flows.

The basic rule:

> Every field and enum value should be justified by at least one system story,
> or explicitly excluded from the current scope.

This repo is a public-safe demo. It uses a small synthetic SaaS workspace
system with real SQL migrations and a real PostgreSQL introspection tool —
the same approach used in production systems.

## What This Demonstrates

- Real SQL migrations as the single source of truth for the schema.
- A registry generated by applying migrations to a disposable PostgreSQL
  database and querying `pg_catalog` — no SQL parsing, no hand-maintained files.
- One Markdown file per business flow.
- A strict `json system-story` block inside each flow file.
- A coverage report showing which schema IDs are covered, deferred, uncovered,
  unknown, or incorrectly categorized.
- Scripts and tests that run in CI against a real Postgres container.

## Example Flow Fragment

The flow block uses a fenced code block with the `json system-story` info
string. Note: most Markdown renderers (including GitHub) do not apply JSON
syntax highlighting to this block because the info string is not a standard
language tag. The content is still valid JSON and is machine-readable by the
coverage tool.

```json system-story
{
  "id": "P1-F02",
  "phase": "P1",
  "title": "Invite admin user",
  "actor": "workspace_owner",
  "actor_sets": [
    "T.user_account.email",
    "T.user_account.full_name"
  ],
  "app_sets": [
    "T.user_account.workspace_id",
    "T.user_account.role",
    "E.user_role.admin"
  ],
  "db_sets": [
    "T.user_account.id",
    "T.user_account.status",
    "E.account_status.invited"
  ],
  "reads": [
    "T.workspace.id",
    "T.workspace.status",
    "E.workspace_status.active"
  ],
  "deferred": []
}
```

```text

```

## Prerequisites

- Python 3.11+
- PostgreSQL 13+ with `psql` on your PATH (or pass `--psql /path/to/psql`)

## Run The Demo

Generate the schema registry from the SQL migrations:

```bash
python tools/system_story/extract_schema.py \
  --migrations-dir examples/workspace-saas/schema/migrations \
  --out-dir        examples/workspace-saas/system-story \
  --host localhost --port 5432 --user postgres --password postgres
```

The script creates a disposable database, applies the migrations, introspects
`pg_catalog`, writes the registry, and drops the database.

Generate the coverage report:

```bash
python tools/system_story/coverage_report.py \
  --registry examples/workspace-saas/system-story/schema-registry.generated.json \
  --flows    examples/workspace-saas/system-story/flows \
  --report   examples/workspace-saas/system-story/coverage-report.generated.md \
  --strict
```

Run tests:

```bash
python -m pip install pytest
python -m pytest
```

## CI

The GitHub Actions workflow starts a Postgres 16 container, runs both tool
scripts, then runs pytest. All three must pass before a commit is green.

## Why It Is Useful

System Story Coverage helps answer:

- Why does this field exist?
- Who sets it: actor, application, or database?
- Which flows read it?
- Which API endpoint or screen needs it?
- Which tests should protect it?
- What changes if this field changes?

It does not replace user stories, BDD, event storming, or OpenAPI. It connects
them to the schema.
